
### 一、 七大核心用例 (Use Cases)

#### 1. 代码理解 (Code understanding)

- **应用场景**：在工程师入职新项目、进行代码调试或调查线上事故时，Codex 帮助他们快速熟悉代码库中陌生的部分。
    
- **具体作用**：工程师使用它来定位功能的关键逻辑，梳理服务或模块之间的关系，并追踪系统中的数据流。此外，它还能挖掘出架构模式或填补缺失的文档（这在平时往往需要耗费大量人工）。在响应突发事件时，Codex 通过展示组件间的交互或追踪故障状态的传播路径，帮助工程师迅速切入新领域。
    
- **工程师真实反馈**：
    
    - **检索系统性能工程师**：在修复 Bug 时，会使用 Codex 的“提问模式 (Ask mode)”来查找代码库中可能出现相同问题的其他地方。
        
    - **API 平台 SRE（网站可靠性工程师）**：在值班时，直接粘贴错误堆栈信息，询问 Codex 认证流程在哪里，它能直接跳转到正确的文件，帮助快速完成故障分类。
        
    - **基础设施服务 DevOps 工程师**：Codex 能跨越 Terraform 和 Python 等不同语言环境，极快地解答“我该在哪里执行此操作？”的问题，速度远超 `grep` 搜索。
        
- **实用提示词 (Prompts) 示例**：
    
    - 这个代码库中的身份验证逻辑是在哪里实现的？
        
    - 总结请求是如何从入口点到响应流经这个服务的。
        
    - 哪些模块与 [插入模块名称] 交互，以及是如何处理故障的？
        

#### 2. 重构与迁移 (Refactoring and migrations)

- **应用场景**：常用于处理跨越多个文件或包的代码变更。例如更新 API、更改某种设计模式的实现，或迁移到新的依赖项。
    
- **具体作用**：当同样的更新需要在几十个文件中进行，或者这种更新依赖于复杂的代码结构且无法通过简单的正则表达式或“查找/替换”完成时，Codex 能确保变更完美一致地执行。它还常用于代码清理，如拆分超大模块、用现代模式替换旧模式、或为提高代码可测试性重构代码。
    
- **工程师真实反馈**：
    
    - **ChatGPT Web 后端工程师**：Codex 将代码中每一个旧的 `getUserById()` 替换为了新的服务模式并直接开启了 PR（拉取请求），几分钟就干完了原本需要几小时的活。
        
    - **ChatGPT Enterprise 产品工程师**：为了清除项目发布的阻碍，让 Codex 扫描每个旧模式的实例，用 Markdown 总结其影响，然后直接开启包含修复代码的 PR。
        
- **实用提示词 (Prompts) 示例**：
    
    - 按关注点将此文件拆分为独立的模块，并为每个模块生成测试。
        
    - 将所有基于回调 (callback-based) 的数据库访问转换为 async/await 异步模式。
        

#### 3. 性能优化 (Performance optimization)

- **应用场景**：识别和解决性能瓶颈。
    
- **具体作用**：在调优或提升可靠性时，工程师通过提示词让 Codex 分析缓慢或内存密集型的代码路径（例如低效的循环、冗余操作或昂贵的查询），并提出优化建议，从而获得可观的效率提升。它还能识别仍在活动使用中的高风险或已弃用的代码模式，帮助团队减少长期技术债务并主动防止性能退化。
    
- **工程师真实反馈**：
    
    - **API 可靠性基础设施工程师**：使用 Codex 扫描重复的昂贵数据库调用，它非常擅长标记热点路径并草拟出可供后续调优的批量查询代码。
        
    - **模型服务平台工程师**：Codex 非常适合快速发现性能问题——花 5 分钟写提示词，能省下 30 分钟的排查工作。
        
- **实用提示词 (Prompts) 示例**：
    
    - 为了提高内存效率优化这个循环，并解释为什么你的版本更快。
        
    - 在此请求处理程序中查找重复的昂贵操作，并建议可以缓存的地方。
        
    - 建议一种更快的方法来批量处理此函数中的数据库查询。
        

#### 4. 提高测试覆盖率 (Improving test coverage)

- **应用场景**：帮助工程师更快速地编写测试代码，特别是在测试覆盖率薄弱或完全缺失的重灾区。
    
- **具体作用**：在修复 Bug 或重构时，让 Codex 建议能覆盖边缘情况或潜在故障路径的测试用例。对于新编写的代码，它可以根据函数签名和周围逻辑自动生成单元测试或集成测试。它特别擅长识别那些人类容易在初始测试中遗漏的边界条件（如空输入、最大长度限制或异常但有效的状态）。
    
- **工程师真实反馈**：
    
    - **ChatGPT Desktop 前端工程师**：在夜间让 Codex 针对低覆盖率模块运行，第二天一早醒来就能看到可运行的单元测试 PR。
        
    - **支付与计费后端工程师**：当切换单体仓库 (mono-repo) 分支令人痛苦时，让 Codex 在后台编写测试并触发 CI，而自己可以继续在当前分支专注工作。
        
- **实用提示词 (Prompts) 示例**：
    
    - 编写此函数的单元测试，包括边缘情况和故障路径。
        
    - 为此排序工具生成基于属性的测试 (property-based test)。
        
    - 扩展此测试文件，以涵盖有关 null 输入和无效状态的缺失测试场景。
        

#### 5. 提升开发速度 (Increasing development velocity)

- **应用场景**：加速开发周期的起点和终点，帮助团队敏捷冲刺。
    
- **具体作用**：在启动新功能时，利用 Codex 搭建脚手架样板代码（生成文件夹、模块和 API 存根），无需人工硬编码每一部分即可快速获得可跑通的代码框架。在项目临近发布面临死线时，Codex 可以分担较小但极其重要的琐事，如分类 Bug、填补最后一公里的代码空白、生成部署脚本、埋点代码 (telemetry hooks) 或配置文件。此外，它还能将产品需求反馈直接转化为初始草稿代码供后续精修。
    
- **工程师真实反馈**：
    
    - **ChatGPT Enterprise 产品工程师**：开了一整天的会，但还是合并了 4 个 PR，因为 Codex 一直在后台代替我敲代码。
        
    - **内部工具全栈工程师**：Codex 完美地帮助交付了 3-4 个原本会在待办清单里烂掉的低优先级修复，让人充满了掌控感。
        
- **实用提示词 (Prompts) 示例**：
    
    - 搭建一个包含基本验证和日志记录的 `POST /events` 新 API 路由。
        
    - 使用此模板 [插入你的遥测代码示例] 生成一个遥测钩子，以跟踪新用户引导流程的成功/失败情况。
        
    - 根据此规范创建代码存根实现：[插入规范或产品反馈]。
        

#### 6. 保持心流状态 (Staying in flow)

- **应用场景**：当工程师的日程变得碎片化、被各种会议或突发打断时，帮助他们维持生产力。
    
- **具体作用**：用于捕获未完成的思路、将笔记转换为工作原型、或剥离出可以稍后重新审视的探索性任务。这使得工程师可以在不丢失思路上下文的情况下暂停和恢复手头工作（特别是当他们值班或高频开会时）。
    
- **工程师真实反馈**：
    
    - **ChatGPT API 后端工程师**：如果发现一个顺手能修复的问题 (drive-by fix)，会丢一个任务给 Codex 去做，而不是中断当前思路去切换分支，等空闲时再去审查 Codex 生成的 PR。
        
    - **基础设施可观测性 API 工程师**：经常将 Slack 聊天串、Datadog 追踪数据、Issue 直接转发给 Codex，这样自己就能保持对最高优先级工作的绝对专注。
        
- **实用提示词 (Prompts) 示例**：
    
    - 生成一个重构计划，将此服务拆分为更小的模块。
        
    - 搭出重试逻辑的存根 (stub)，并添加一个 TODO 注释，我稍后会来填补退避逻辑 (backoff logic)。
        
    - 总结一下这个文件，以便我明天能接着今天离开的地方继续写。
        

#### 7. 探索与构思 (Exploration and ideation)

- **应用场景**：用于开放式的发散工作，如寻找替代解决方案或验证技术设计决策。
    
- **具体作用**：可以让 Codex 提供解决问题的不同路线、探索团队不熟悉的代码模式或对当前假设进行压力测试。这有助于直观地展现技术权衡、扩展设计选项并锐化实现细节。它还能用于“顺藤摸瓜”：给定一个已知 Bug 或已弃用的方法，Codex 能扫描代码库识别类似模式，将隐藏的回归错误连根拔起。
    
- **工程师真实反馈**：
    
    - **ChatGPT Desktop 产品工程师**：Codex 帮助我解决了项目冷启动问题——粘贴产品规范和文档，它就能搭好代码骨架并指出我思维中的盲区。
        
    - **检索系统性能工程师**：修复 Bug 后，我会问 Codex 哪里可能还潜伏着类似问题，并顺势派发后续的清理任务。
        
- **实用提示词 (Prompts) 示例**：
    
    - 如果这个系统变成事件驱动的，而不是现在的请求/响应模式，代码将如何运作？
        
    - 找到所有手动拼接 SQL 字符串而不是使用我们查询构建器的模块。
        
    - 以更偏向函数式的编程风格重写此代码，避免变量突变 (mutation) 和副作用。

### 二、 最佳实践 (Best Practices)

要让 Codex 稳定输出高质量代码，OpenAI 团队总结出了以下极具价值的高阶心法：

1. **从“提问模式 (Ask Mode)”起步**：对于量级较大的变更，首要步骤是使用 Ask 模式提示 Codex 生成一份“实施计划 (implementation plan)”，然后再将计划作为输入去切换到“代码模式 (Code Mode)”。这种“先定框架，再写代码”的双步流能有效约束 Codex，避免逻辑跑偏跑错。目前它最适合处理人类需要一小时左右完成、或几百行规模的任务（随着模型进化，处理上限会进一步拉升）。
    
2. **迭代优化 Codex 的开发环境**：为 Codex 配置好启动脚本、环境变量并授予其互联网访问权限，能断崖式降低其报错率。持续把构建错误通过环境配置纠正给 AI 学习，虽然前期需要磨合迭代，但长远来看带来的效率红利是巨大的。
    
3. **像撰写 Github Issue 一样构建 Prompt**：AI 需要极强的信息密度。当你的提示词包含文件路径、组件名称、代码 Diff 对比以及相关文档片段时，Codex 的响应最为精准。使用诸如“请采用与 [模块 X] 相同的实现方式”这样的 Few-shot 模式效果奇佳。
    
4. **将 Codex 任务队列作为轻量级 Backlog（待办清单）**：不要强求一次生成完美的庞大 PR。把那些切线想法、残缺的工作丢给 Codex 排队处理即可。Codex 是一个绝佳的“工作暂存区”，等你恢复专注力时再回来验收成果即可。
    
5. **使用 `AGENTS.md` 文件喂养持久上下文**：强烈建议在代码仓库中维护一个专属的 `AGENTS.md` 文件。在这个文件里写入你们的命名约定、业务逻辑特例、历史遗留怪癖或深层依赖项，Codex 会跨越提示词去读取这些无法单凭代码推断出的“潜规则”，从而更丝滑地融入你的项目。
    
6. **利用“N中选优 (Best of N)”机制**：对于复杂任务，开启 Best-of-N 特性同时生成多套不同的代码解法，你可以借此快速发散思维，甚至从不同方案中“缝合”出最优的代码骨架。

### 三、 展望未来 (Looking Ahead)

文档在结语中提到，尽管 Codex 仍处于研究预览状态，但它已经实质性地颠覆了 OpenAI 内部开发软件的节奏：帮助团队动作更快、代码质量更高，且有余力去处理那些曾经由于资源紧张而无限期搁置的任务。随着模型本身的进化和与工作流的深度融合，OpenAI 期待在未来解锁出更加震撼的软件开发方式。
